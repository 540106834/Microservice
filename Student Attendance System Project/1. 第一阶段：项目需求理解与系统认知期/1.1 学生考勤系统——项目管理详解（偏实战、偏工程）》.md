# 考勤项目管理详解

> 一句话定调：
> **考勤系统不是技术难，而是“人多、角色多、边界多”。**

---

## 一、项目管理的核心目标

在这个项目里，管理目标只有四个字：

> **可控、可交付**

拆开就是：

* 需求可控（不天天变）
* 进度可控（不靠加班）
* 质量可控（不靠祈祷）
* 风险可控（提前兜底）

---

## 二、项目角色与职责划分

### 1️⃣ 项目核心角色

| 角色       | 主要职责     |
| -------- | -------- |
| 产品 / 需求方 | 提规则、提口径  |
| 项目负责人    | 排期、拍板、兜底 |
| 开发       | 实现功能     |
| 测试       | 找问题      |
| 运维       | 部署、稳定性   |
| 学校管理员    | 实际使用者    |

⚠️ 在考勤系统里，**学校管理员和班主任是隐藏关键人**。

---

## 三、需求管理（最容易翻车的地方）

### 1️⃣ 需求来源分类

* 政策类（必须做）
* 管理类（统计、报表）
* 使用体验类（老师提的）

👉 管理策略：**优先级不一样**

---

### 2️⃣ 需求拆解方式

不要写成：

> “实现学生考勤功能”

要拆成：

* 学生上学打卡
* 放学打卡
* 迟到规则配置
* 考勤结果查询

📌 **拆到“可以估工期”为止**

---

### 3️⃣ 需求冻结点

* 数据模型冻结
* 编码规则冻结
* 考勤规则冻结

> 一旦冻结，改就是“新需求”，不是“顺手改一下”。

---

## 四、计划与里程碑管理

### 阶段划分（推荐）

| 阶段   | 可交付物    |
| ---- | ------- |
| 需求确认 | 需求说明书   |
| 设计阶段 | 架构图、表设计 |
| 开发阶段 | 可跑系统    |
| 测试阶段 | 测试报告    |
| 上线准备 | 部署文档    |
| 正式运行 | 监控指标    |

📍 每个阶段都要**有文件、有结果**。

---

## 五、范围管理（防止项目膨胀）

### 典型“范围膨胀”话术

* “顺便加一个功能”
* “其实很简单”
* “别的学校也要”

🎯 应对策略：

* 所有新增需求走变更单
* 明确影响：

  * 工期
  * 风险
  * 数据结构

---

## 六、进度管理（不靠催）

### 1️⃣ 任务拆解原则

* 一个任务 ≤ 2 天
* 能独立验收
* 有明确完成标准

### 2️⃣ 进度跟踪方式

* 每日 15 分钟同步
* 看**阻塞点**，不是看谁忙

---

## 七、质量管理（不是测试一个人的事）

### 质量控制点

| 阶段 | 控制点       |
| -- | --------- |
| 设计 | 表结构评审     |
| 开发 | 代码 Review |
| 测试 | 场景覆盖      |
| 上线 | 回滚方案      |

📌 **考勤系统最怕“边界情况”**：

* 请假
* 节假日
* 补卡

---

## 八、风险管理（提前兜底）

### 常见风险清单

* 政策临时调整
* 学校差异太大
* 打卡设备不稳定
* 数据丢失

### 对策示例

* 规则配置化
* 日志留痕
* 手工补录入口
* 定期数据备份

---

## 九、沟通管理（比写代码重要）

### 沟通对象与重点

| 对象 | 关注点   |
| -- | ----- |
| 领导 | 进度、风险 |
| 老师 | 好不好用  |
| 运维 | 稳不稳定  |
| 开发 | 需求清不清 |

🎯 原则：
**用对方听得懂的话沟通**

---

## 十、上线与运行管理

### 上线前 Checklist

* 数据初始化完成
* 管理员账号准备好
* 回滚方案演练
* 高峰期压力评估

### 运行期关注指标

* 打卡成功率
* 高峰响应时间
* 异常考勤比例

---

## 十一、项目复盘（成长最快的环节）

复盘关注三件事：

1. 哪些设计救了项目
2. 哪些地方差点翻车
3. 下次可以提前做什么

📘 **复盘文档比代码值钱**。
